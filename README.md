## 🛠️ Performance & Principles

We prioritize:

- **Clean Architecture**: Separation of concerns and modularity.
- **Explicit Trade-offs**: Architecture is about optimizing under constraints.
- **Scalability**: Designing for growth, not just for today.

## 🤝 How to Contribute

We welcome contributions! If you have a topic you'd like to add or an improvement to existing documentation, please follow these steps:

1. **Fork the Repository**: Create your own copy of the project.
2. **Create a Feature Branch**:
   ```bash
   git checkout -b feature/amazing-new-topic
   ```
3. **Commit Your Changes**: Follow our naming conventions and write clear messages.
4. **Open a Pull Request**: Submit your changes for review.

### Contribution Guidelines:

- **Project Structure**: Ensure new files are placed in the correct category (`Frontend/` or `Backend/`).
- **Naming Conventions**: Use `Kebab-Case` for filenames and consistent heading hierarchies.
- **Commit Messages**: Use the imperative mood and follow the [Conventional Commits](https://www.conventionalcommits.org/) format if possible.
  - _Example_: `feat: add guide for message broker patterns`

- **Content Quality**: Provide real-world examples and discuss trade-offs (Pros/Cons).
