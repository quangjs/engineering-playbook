## 1. Cross-Site Scripting (XSS)

XSS is the "bread and butter" of front-end vulnerabilities. It occurs when an attacker injects malicious scripts into content that other users see.

Example:

```html
<script>
  alert("XSS");
</script>
```

- **Stored XSS:** The script is permanently stored on the server (e.g., in a database via a comment field).
- **Reflected XSS:** The script is "reflected" off a web server, usually via a URL parameter or search query.
- **DOM-based XSS:** The vulnerability exists entirely in the client-side code (e.g., using `innerHTML` to render a username from a URL fragment).

**Prevention Strategies:**

- **Sanitization:** Use libraries like `DOMPurify` to clean HTML.
- **Escaping:** Convert special characters to HTML entities (e.g., `<` becomes `&lt;`).
- **Use Modern Frameworks:** React and Vue automatically escape content in data bindings, but beware of "escape hatches" like `dangerouslySetInnerHTML`.

## 2. Content Security Policy (CSP)

A CSP is your second line of defense. It’s an HTTP response header that tells the browser which sources of content (scripts, styles, images) are trusted.

Example:

```html
Content-Security-Policy: default-src 'self'; script-src 'self'
https://trusted.com;
```

- **How it works:** If an attacker manages to inject a script tag, a strong CSP will prevent it from executing if the source isn't on the "allow-list."
- **Key Directive:** `script-src 'self' https://trusted.com;`. This prevents inline scripts and scripts from unknown domains.

## 3. Cross-Site Request Forgery (CSRF)

CSRF tricks a logged-in user into submitting a malicious request to a different website where they are authenticated.

- **The Scenario:** You are logged into `bank.com`. You visit `evil.com`, which contains a hidden form that sends a POST request to `bank.com/transfer`. Since your browser automatically attaches your session cookies, the bank thinks _you_ made the request.

**Prevention Strategies:**

- **Anti-CSRF Tokens:** A unique, secret token generated by the server and sent to the client. The client must send it back with any state-changing request.
- **SameSite Cookie Attribute:** \* `SameSite=Strict`: Cookies only sent on first-party requests.
- `SameSite=Lax`: (Default in modern browsers) Cookies aren't sent on cross-site subrequests (like images) but are sent when navigating to the site.

## 4. Secure Storage & Sensitive Data

Where you store data in the browser matters.

| Storage Type       | Accessibility                                   | Security Risk                                               |
| ------------------ | ----------------------------------------------- | ----------------------------------------------------------- |
| **LocalStorage**   | Accessible by any script on the same origin.    | Highly vulnerable to **XSS**. Never store JWTs or PII here. |
| **SessionStorage** | Same as LocalStorage, but cleared on tab close. | Also vulnerable to XSS.                                     |
| **Cookies**        | Can be restricted via flags.                    | Vulnerable to CSRF and XSS (unless flagged).                |

**The "Gold Standard" for Tokens:**
Store sensitive tokens in an **HttpOnly** and **Secure** cookie. This prevents JavaScript from reading the token, neutralizing XSS-based theft.

---

## 5. Security Headers to Know

Beyond CSP, these headers are low-hanging fruit for interviewers:

- **Strict-Transport-Security (HSTS):** Forces the browser to communicate over HTTPS only.
- **X-Content-Type-Options: nosniff:** Prevents the browser from trying to guess the "MIME type" of a file, which can lead to script execution.
- **X-Frame-Options:** Prevents your site from being put in an `<iframe>`, which stops **Clickjacking** attacks.
