## 1. Git Branching Strategies

A branching strategy defines how a team uses Git to deliver code. The two most popular methods are **Trunk-based Development** and **Gitflow**.

### A. Trunk-based Development (The Modern Standard)

In this model, all developers work on a single branch (the "trunk," usually `main`). They push small, frequent updates.

- **How it works:** Developers pull from `main`, make a quick change in a short-lived branch, and merge it back as soon as possible.
- **Key Requirement:** **Feature Flags**. Since code is merged quickly, you use flags to hide "work-in-progress" features from users in production.
- **Pros:** High velocity, eliminates "merge hell," perfect for **CI/CD**.
- **Cons:** Requires high senior oversight and excellent automated testing, as a mistake hits `main` immediately.

### B. Gitflow (The Traditional Model)

Gitflow is a strict, scheduled branching model designed for products with a specific release cycle (e.g., version 1.0, 2.0).

- **The Branches:**
- `main`: Always contains production-ready code.
- `develop`: The integration branch for features.
- `feature/*`: Temporary branches for new features.
- `release/*`: Preparation for a new production release.
- `hotfix/*`: Quick fixes for production bugs.

- **Pros:** Very organized, clear separation of "work-in-progress" vs. "ready."
- **Cons:** Slow, complex, and prone to massive "merge conflicts" when merging `develop` into `main` after a long period.

## 2. Advanced Git Commands to Know

You should be able to explain these three commands and when to use them:

### Rebase vs. Merge

- **Merge:** Combines two branches and creates a "merge commit." It preserves the history of exactly when things happened.
- **Rebase:** Moves your branch's starting point to the tip of another branch. It creates a **linear history**.
- **Interview Tip:** "I use `rebase` on my local feature branch to keep it up-to-date with `main` without creating messy merge commits. I use `merge` when I'm officially moving code into a shared branch like `develop` or `main`."

### Squash and Merge

This takes all the commits in your feature branch (e.g., "fix typo," "add button," "fix typo again") and combines them into **one single, clean commit** before merging into `main`. This keeps the `main` history very readable.

### Git Cherry-pick

Allows you to pick **one specific commit** from one branch and apply it to another.

- **Scenario:** You fixed a critical bug in a "Feature A" branch, but you need that fix in "Production" right now without bringing the rest of Feature A with it.

## 3. Git Best Practices for Front-End Teams

- **Atomic Commits:** One commit should do one thing. Don't mix "Fixing a bug" and "Refactoring CSS" in the same commit.
- **Descriptive Messages:** Use the **Imperative Mood** (e.g., "Add login validation" instead of "Added some stuff to login").
- **Git Hooks (Husky):** Automatically run `npm test` or `lint` before a commit is allowed to be created. This prevents "bad code" from ever leaving a developer's machine.

## 4. Comparison Summary

| Feature               | Trunk-based                      | Gitflow                          |
| --------------------- | -------------------------------- | -------------------------------- |
| **Release Frequency** | Constant (Multiple times a day). | Scheduled (Weekly, Monthly).     |
| **Branch Lifespan**   | Very short (hours/days).         | Long (weeks).                    |
| **Complexity**        | Low (few branches).              | High (many branches).            |
| **Ideal For**         | SaaS, Startups, CI/CD.           | Enterprise, Legacy, Mobile Apps. |
