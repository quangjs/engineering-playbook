## 1. async vs defer vs normal

| Attribute   | Parsing HTML | Fetching Script         | Executing Script                        |
| ----------- | ------------ | ----------------------- | --------------------------------------- |
| **Normal**  | 🛑 Pauses    | 🌐 Downloads            | ⚙️ Executes immediately                 |
| **`async`** | ✅ Continues | 🌐 Downloads (parallel) | 🛑 Pauses parsing to execute            |
| **`defer`** | ✅ Continues | 🌐 Downloads (parallel) | 🕒 Executes only **after** DOM is ready |

---

Think of performance optimization not as a single task, but as a **chain of events**. If one link is slow, the whole experience feels sluggish. In an interview, you'll stand out if you categorize optimizations by **where** they happen in the lifecycle of a page.

---

## 2. The Network Level (Fast Delivery)

Before the browser can even read your code, it has to fetch it.

- **Compression:** Use **Brotli** instead of Gzip. It’s the modern standard and offers 15-20% better compression.
- **HTTP/2 & HTTP/3:** These allow **Multiplexing**, meaning the browser can download many files over a single connection simultaneously, rather than waiting for one to finish before starting the next.
- **CDN (Content Delivery Network):** Serve static assets from a server physically close to the user to reduce latency.
- **Caching Strategy:** Use `Cache-Control: max-age=31536000, immutable` for hashed files (e.g., `main.a1b2c3.js`).
- **Minification & Bundling:** Use tools like Webpack or Vite to bundle and minify your assets.
- **Tree Shaking:** Remove unused code from your bundle.

---

## 3. HTML & The Critical Rendering Path (CRP)

This is about how the browser turns code into pixels.

- **Minimize DOM Depth:** A deep DOM tree makes calculating styles and layouts expensive. Keep it flat.
- **Resource Hints:** Tell the browser what's coming:
- `<link rel="preload">`: For high-priority assets (Fonts, Hero images).
- `<link rel="preconnect">`: For 3rd party domains (API endpoints, Google Fonts).

- **Eliminate Render-Blocking:** Put CSS in the `<head>` and use `defer` for scripts to ensure the HTML parser isn't interrupted.

---

## 4. Image & Media Optimization

Images usually account for the bulk of a page's weight.

- **Modern Formats:** Use **WebP** or **AVIF**. Avoid PNG/JPG unless necessary.
- **SVG:** Perfect for logos, icons, and simple illustrations.
  - **Pros:** Infinite scalability, tiny file size, CSS-stylable.
  - **Cons:** High DOM complexity if the path is too detailed (causing lag).
- **Responsive Images:** Use the `<picture>` tag or `srcset` attribute. Don't send a 4000px image to a 400px phone screen.
- **Native Lazy Loading:** Use `loading="lazy"` for any image "below the fold" (not visible on the initial screen).
- **Content Layout Shift (CLS):** Always provide `width` and `height` attributes to images so the browser can reserve space before the image loads, preventing the page from "jumping."

---

## 4. JavaScript & Frameworks (React/Next.js)

JS is the most expensive resource because the browser has to download, parse, compile, **and** execute it.

### General JS

- **Tree Shaking:** Ensure your build tool (Webpack/Vite) removes unused code. Use ES Modules (`import/export`) to make this easier.
- **Code Splitting:** Don't ship the whole app at once. Use dynamic imports:

```javascript
// Instead of: import { AdminPanel } from './Admin';
const AdminPanel = React.lazy(() => import("./Admin"));
```

### React Specifics

- **Memoization:** Use `React.memo` for components, and `useMemo` / `useCallback` for expensive calculations or stable function references.
- **Virtualization:** For long lists (1000+ items), use libraries like `react-window`. Only the items visible on the screen are rendered in the DOM.

### Next.js

- **Automatic Image Optimization:** The `next/image` component handles resizing, lazy loading, and WebP conversion automatically.
- **Incremental Static Regeneration (ISR):** Updates static content _after_ the site is built, giving you the speed of static sites with the freshness of dynamic data.
- **Server Components:** In Next.js 13+, code in Server Components never reaches the client's browser, drastically reducing the JS bundle size.

## 5. Measuring Success (Core Web Vitals)

Interviewer: _"How do you know if your optimization worked?"_
Answer: _"By monitoring the Core Web Vitals."_

| Metric  | Name                      | Goal                                                   |
| ------- | ------------------------- | ------------------------------------------------------ |
| **LCP** | Largest Contentful Paint  | Under **2.5s** (How fast the main content appears)     |
| **INP** | Interaction to Next Paint | Under **200ms** (How snappy the UI feels when clicked) |
| **CLS** | Cumulative Layout Shift   | Under **0.1** (How much the page "jumps" during load)  |

---

## Interview "Cheat Sheet" Summary

If asked for a "quick win" strategy, give them this **3-Step Audit**:

1. **Analyze:** Use Chrome DevTools (Lighthouse) or WebPageTest to find the bottleneck.
2. **Optimize Delivery:** Enable Brotli, move to a CDN, and implement `next/image`.
3. **Optimize Execution:** Code-split the JS bundles and defer non-critical scripts.
